# Death Mechanic

The Cabin has two terminal fail states. Both end the run with a single authored line — no "you died" screen, no menu, no fourth-wall break. This is the implementation side of the **Death & Failure** entry in `mechanics.md`.

## Overview

The player has two stats: `health` (0–100) and `fear` (0–100). Crossing a threshold ends the run.

| Stat   | Threshold       | Closing line                  | Meaning              |
| ------ | --------------- | ----------------------------- | -------------------- |
| `fear` | `>= 100`        | _The dark consumes you._      | Psychological collapse |
| `health` | `<= 0`        | _The dark watches you fade._  | Physical death       |

If both thresholds land in the same turn, **fear collapse wins** — the mind goes before the body.

## Behaviour

- The check fires after any path that mutates `fear` or `health`:
  - The action/effect/event path in `handle_user_input` (covers AI-suggested effects, action results, wildlife attacks, and `thrown_into_darkness`).
  - The `LOAD` branch (a save persisted at the threshold ends the run on load).
- Pending action feedback prints **before** the closing line so the last beat lands the right way around.
- After the line lands, the run loop exits cleanly. The next prompt is never shown.

## Story integration

- The Act II Lyer encounter (`Map._trigger_lyer_encounter`) clamps `fear` short of 100 (`min(99, …)`) and `health` above 0 (`max(1, …)`). The beat is meant to wound, not kill — letting it kill would end the run mid-scene before the layer flip lands.
- The closing lines are authored canon. They are not generated by the AI interpreter and must not be rewritten as flavour. The Lyer is implied — never named, never explained in mechanics.

## Code

- `game/death.py` — shared module. Owns the closing-line constants and `death_line_for(player)`, which encodes the thresholds and the precedence rule. Both `GameEngine` (terminal) and `WebGameSession` (browser) call it, so both surfaces end the run on identical terms.
- `GameEngine._check_death()` — terminal-side render. Flushes pending feedback, prints the line, sets `running = False`.
- `WebGameSession._handle_play_input()` — browser-side render. Sets phase to `ENDED` and emits a `RenderFrame` with `game_over=True` and the closing line.

## Anti-patterns

- **No "Game Over" or "You died" text.** Failures are narrated, not labelled.
- **No load/quit menu after death.** The closing line is the last text the run ever prints.
- **No telegraphing.** Fear at 99 reads identically to fear at 80. The closing line is the only signal.
- **No different lines for different surfaces.** Terminal and browser must stay in sync via `game/death.py`.

## Deferred

The following are out of scope for the current implementation; flag tickets if they become wanted:

- **Wrong-layer variants.** The mechanic is layer-blind today. The wrong cabin and wrong woods may eventually want bespoke closing beats — kept open for a future pass.
- **Save / quit / auto-load after death.** A post-death recovery menu belongs to a save UX pass, not the death mechanic.
- **Pre-death telegraphing.** Visual or narrative cues as the player approaches a threshold. Open question whether this is worth the cost — abruptness may be part of the horror.
